# Catalog — Codebase Review
**Date:** 2026-06-22  
**Branch reviewed:** `main` (tip: `3131c45`)  
**Scope:** Consumer app, admin panel, shared services, CSS

---

## Executive Summary

The consumer app is architecturally sound — lazy-loaded overlays, pooled video playback via TrailVideoHost, IntersectionObserver-driven card activation, and a well-thought-out iOS Safari toolbar fix all show a mature, production-tuned codebase. The biggest risks now are **scale and maintainability** rather than correctness.

The single critical concern: `app/routes/_index.tsx` has grown to **2,902 lines** and owns ~50+ `useState` hooks, 40+ event handlers, and 10+ async service calls simultaneously. This file is a change-collision magnet and the hardest file in the codebase to onboard onto. All other issues radiate outward from this one.

Beyond that, three categories of improvements stand out: **missing resilience** (no error boundaries, unguarded localStorage, silent async failures), **performance hotspots** that have already been identified as risks but not yet addressed (O(n²) particle canvas, multiple unthrottled scroll listeners, per-keystroke search calls), and **accessibility gaps** (no ARIA on interactive controls).

## UI Improvements

### UI-1 — Feed grid has no skeleton while loading (`GridView.tsx`)

When the Supabase `getLooks()` call is in-flight, the grid container renders empty — there are no placeholder/shimmer cards. The shimmer CSS animation already exists in `grid-view.css` (`.shimmer-card`), but there's no conditional code path that renders shimmer tiles while `looks.length === 0` and the fetch hasn't resolved yet. On a slow connection (or cold edge-function start) the user sees a blank dark screen for 1–3 seconds with no visual indication that content is coming.

**Suggestion:** Render 12 `.shimmer-card` placeholder tiles while `loading === true`, removing them once the first real page of looks arrives.

---

### UI-2 — Glass bottom bar bleeds on light hero background (`bottom-bar.css`, line 59)

The bottom bar uses a glass gradient: `rgba(255,255,255,0.14)` at the top fading to a darker base. On the dark feed this is invisible and looks correct. But when the `ShoppingForHero` ceremony or the auth splash is visible (light/gradient backgrounds), the glass pill can become invisible because there's no sufficient contrast between the bar's frosted surface and the background.

**Suggestion:** Conditionally add a `box-shadow: 0 0 0 1px rgba(255,255,255,0.12)` border when the hero is active (via a CSS class on `.app-root.hero-mode`) or use `backdrop-filter: brightness(0.8)` to guarantee minimum contrast in all backgrounds.

---

### UI-3 — Look card context menu has no visual polish on dark backgrounds (`LookCard.tsx`, lines 375–388)

The super-admin long-press context menu is styled entirely with inline styles directly in JSX. It uses a hardcoded `background: 'rgba(0,0,0,0.85)'` that doesn't track the app's `--bg` variable, and has no separation from the look thumbnail behind it — the card's video can bleed through on older devices that don't composite backdrop filters.

**Suggestion:** Move the context menu styles into a `look-card-context-menu.css` file that references `var(--bg)` and `var(--text)`. Add `border: 1px solid rgba(255,255,255,0.1)` for separation.

---

### UI-4 — Typography scale is inconsistent across CSS files

Font sizes are set independently in each CSS file with raw `px` values (`12px`, `13px`, `14px`, `15px`, `16px`) rather than referencing a shared scale. This results in subtly mismatched sizes between e.g. the bottom bar suggestion chips (`12px`), the look card creator label (`13px`), and the overlay product name (`15px`). There is no CSS custom property for a type scale (e.g. `--text-xs: 12px`, `--text-sm: 13px`).

**Suggestion:** Declare a type scale in `:root` inside `base.css` and replace all raw font-size values throughout the CSS files. This also makes global scale changes (e.g. for accessibility/larger-text mode) trivially easy.

---

### UI-5 — No empty state on the Bookmarks page (`BookmarksPage.tsx`)

When a user opens Bookmarks for the first time (or has cleared all their saves), the page renders an empty list with no CTA or messaging. There is no illustration, no "nothing saved yet" copy, and no "explore the feed" link to redirect them.

**Suggestion:** Add a centered empty-state block (similar to the existing `EmptyCatalogState` component) for each of the three bookmark tabs (Looks, Products, Brands) when their respective arrays are empty.

## UX Improvements

### UX-1 — Search fires on every keystroke, no debounce (`BottomBar.tsx`, line 321)

`handleSearchInput()` calls `emitSearch()` synchronously on every character the user types. A user typing "black mini dress" triggers 15+ search calls. Even if the network calls resolve in the right order, the UI updates with intermediate-query results while the user is still typing, creating flicker. There's also no `useTransition` to deprioritize the render.

**Suggestion:** Wrap `emitSearch()` in a 250ms debounce. Optionally wrap the downstream state update in `React.startTransition` so mid-type renders don't block the input field from feeling responsive.

---

### UX-2 — Accidental bookmarks and follows have no undo

Tapping the bookmark icon or the follow button immediately commits the action to Supabase with no undo affordance. If a user accidentally saves a product or follows a creator, they must navigate to Bookmarks or the creator's page to reverse it. The `emitSavedToast` utility (`app/utils/savedToast.ts`) already shows a toast — that toast could host an Undo button.

**Suggestion:** Add an "Undo" action to the saved/follow toast that fires the inverse action (unsave/unfollow) within a 4-second window before the toast disappears.

---

### UX-3 — Search doesn't normalize diacritics (`BottomBar.tsx`, lines 355–368)

Type-ahead suggestion matching uses `.startsWith()` + `.includes()` directly on the suggestion strings. Searching for "cafe" will not match a suggestion that contains "café". This is a minor but real gap for any fashion-adjacent search terms that include brand names or foreign words.

**Suggestion:** Normalize both the query and the suggestion strings with `str.normalize('NFD').replace(/\p{Diacritic}/gu, '')` before comparison.

---

### UX-4 — No feedback when an async overlay action fails silently

`prefetchSimilarProducts()` and `getGraphPairs()` failures in `_index.tsx` are caught and silently swallowed. The product rail and similar-looks row simply appear empty with no indication to the user that something went wrong. Similarly, if `handleOpenProduct()` fails to resolve a product ID via URL, the rail is empty with no "retry" affordance.

**Suggestion:** On fetch failure, show a minimal inline error state inside the rail ("Couldn't load similar looks · Retry") backed by a retry callback. Log the failure to a metrics endpoint so these can be monitored.

---

### UX-5 — Filter panel state is lost when the user switches looks

When a user opens a look overlay, then closes it, then opens a different look, the product filter state in the look overlay is reset. If a user was in the middle of comparing filtered results across two looks, they have to re-apply filters every time they switch.

**Suggestion:** Persist the look overlay filter state (selected tags, sort order) in a `useRef` or in the parent's state so it survives overlay-to-overlay navigation. Clear it only on an explicit "reset" action or on a full overlay close.

## Codebase Improvements

### CB-1 — `_index.tsx` is 2,902 lines and must be decomposed

The main route file has accumulated state, routing logic, analytics pings, playback coordination, gesture detection, and overlay management simultaneously. With 50+ `useState` calls and 40+ event handlers in a single component tree, every change risks cross-concern bugs and merge conflicts. Specific refactors to pull out:

- `useLookTrail` — look open/close, nav stack, back-navigation, URL sync
- `useSearchState` — query, filters, catalog name, suggestion pool
- `useChromeVisibility` — header/bottom bar hide/show, scroll listeners, scroll position saving
- `useOverlayCallbacks` — product open, creative open, creator open, brand open handlers
- `usePlaybackCoordinator` — `pauseAllVideos`, hero video resume, video pipeline mode

Each hook would be a file in `app/hooks/` and the route component itself would be a thin assembly of those hooks. The route's JSX (currently ~600 lines) should remain, but the 2,000+ lines of logic should move into these hooks.

---

### CB-2 — Particle canvas uses O(n²) connection-drawing loop (`GridView.tsx`, lines 65–110)

The `ParticleField` component draws connection lines between particles by iterating all pairs: for each particle, it checks the distance to every other particle. With 200 particles, that's 40,000 `Math.sqrt()` calls per animation frame (~60fps = 2.4M calls/sec). On a low-end Android device this tanks the UI thread.

**Suggestion:** Either reduce particle count to ≤50 on mobile (gated by `isMobileViewport()`), or replace the O(n²) loop with a simple grid-binning approach: only check particles in the same and neighboring grid cells. Alternatively, drop connection lines entirely on mobile — the visual difference is minimal.

---

### CB-3 — Multiple competing scroll listeners without rAF batching (`_index.tsx`, lines 564–648)

Two scroll handlers run on the same `.home-feed-wrap` element: one for the hero state transition (lines 564–596) and one for the Chrome hide/show logic (lines 630–648). Neither batches with `requestAnimationFrame`. On a 120Hz display, a fast scroll triggers both handlers 120 times/second. On iOS, each synchronous scroll handler delays the next paint.

**Suggestion:** Consolidate the two handlers into a single `handleFeedScroll` function that reads `scrollTop` once per rAF frame and dispatches to both the hero and chrome logic. Use a `useRef`-backed `isScheduled` guard: only register a new rAF if one isn't already pending.

---

### CB-4 — Magic numbers scattered across components and CSS

Dozens of raw numeric literals are used directly in logic and CSS without named constants: `220` (scroll threshold to show the back button in `LookOverlay.tsx`), `56` (drag-dismiss threshold in `BottomBar.tsx`), `400px` (IntersectionObserver rootMargin in `GridView.tsx`), `8` (scroll delta for the Daily Feed bar detection), `12` (BATCH_SIZE for look fetching).

**Suggestion:** Collect these into a `app/constants/layout.ts` file (or per-domain constants files). CSS magic numbers should become CSS custom properties on `:root` so the admin dials system can eventually control them without code deploys.

---

### CB-5 — `isProductBookmarked()` does a linear scan on every call (`useBookmarks.ts`, ~line 69)

The `isProductBookmarked(key)` function in `useBookmarks` iterates the bookmarked products array and compares keys one-by-one. This function is called on every product rendered in the look overlay's product list. With 20 products visible and 100+ saved bookmarks, that's 2,000 comparisons per render.

**Suggestion:** Maintain a parallel `bookmarkedProductKeys: Set<string>` alongside the array. Update the Set in the same `useState` setter. The `isProductBookmarked` check becomes `O(1)`.

---

### CB-6 — localStorage access is not guarded against quota errors or corruption

`useBookmarks.ts` reads from and writes to `localStorage` without try/catch. On iOS Safari in private browsing mode, `localStorage.setItem` throws a `QuotaExceededError`. If a user has filled their storage quota, every bookmark interaction crashes silently and the UI behaves as if the save succeeded. Similarly, `JSON.parse` on a corrupted value throws, causing the hook to return undefined instead of an empty default.

**Suggestion:** Wrap all `localStorage.getItem` / `setItem` / `JSON.parse` calls in try/catch. On read failure, return the safe default value. On write failure, optionally emit a toast ("Couldn't save — storage full").

---

### CB-7 — Custom event names are untyped string literals

Inter-component communication uses `window.dispatchEvent(new CustomEvent('catalog:close-search'))`, `catalog:look-seen`, `catalog:open-bookmarks`, etc. These string names appear in both the dispatch sites and the listener sites (`addEventListener`), with no shared type or constant. A typo at either end silently breaks the feature.

**Suggestion:** Create `app/constants/events.ts` that exports typed event name constants:

```ts
export const EVENTS = {
  CLOSE_SEARCH: 'catalog:close-search',
  LOOK_SEEN: 'catalog:look-seen',
  OPEN_BOOKMARKS: 'catalog:open-bookmarks',
} as const;
```

This also doubles as documentation of the full event surface, which is important for the Flutter shell integration (Section 8 of CLAUDE.md).

---

### CB-8 — `selectedProductRef` and `selectedProduct` duplicate the same value (`_index.tsx`, lines 372–382)

Several pieces of state are tracked simultaneously as both a `useState` value and a `useRef` (e.g. `selectedProductRef` mirrors `selectedProduct`). The ref exists so async callbacks always read the latest value without stale closures, which is a valid pattern — but the dual-track maintenance means any setter must update both, and a missed update leaves them out of sync.

**Suggestion:** Use a single `useRef` for the "always-current" read, and derive the reactive version with a counter-based trigger where needed. Or adopt Zustand for the shared navigation state, which solves both the stale-closure problem and the ref/state duplication simultaneously.

---

### CB-9 — `manualChunks()` in `vite.config.ts` is fragile and hard to maintain (`vite.config.ts`, lines 82–164)

The manual chunk routing function is 82 lines of `id.includes(...)` conditionals. If a new module is imported by two different chunks' dependency trees, it silently lands in the first matching rule rather than a shared chunk. Adding a new overlay or admin section requires manually editing this function to avoid bloating the consumer bundle.

**Suggestion:** Replace the manual chunk function with Rollup's automatic chunk splitting (`output.manualChunks: undefined`) for the consumer bundle's lazy-loaded overlays — Rollup's algorithm handles shared modules correctly. Keep explicit splitting only for the `admin` vs. `consumer` boundary (these are large, disjoint trees that benefit from a hard split). Add a build-time size-check assertion so CI fails if any entry chunk exceeds a configured limit.

---

### CB-10 — No top-level error boundary; a single thrown exception unmounts the entire app

There is no `<ErrorBoundary>` component wrapping the app tree in `app/root.tsx` or `_index.tsx`. If any component throws during render (e.g., a malformed Supabase response passed to a component that doesn't expect nulls, or a third-party script error), React's default behavior is to unmount the entire tree and display a blank screen with no message.

**Suggestion:** Add a `<CatalogErrorBoundary>` component that catches render errors, logs them to a telemetry endpoint, and renders a minimal "Something went wrong — tap to reload" fallback. Wrap the overlay stack with a separate boundary so a crash inside `LookOverlay` or `ProductPage` doesn't take down the entire feed.
