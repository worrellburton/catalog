## Summary of Biggest Opportunities

1. **Memory leaks in unbounded caches** — the follow-state cache and the TrailVideo pool both grow without eviction as users scroll through large feeds. On long sessions, this accumulates real memory pressure.
2. **Missing error boundaries around nested feeds** — a crash in any `ContinuousFeed` inside `LookOverlay` or `ProductPage` takes down the whole overlay, forcing the user to restart from the grid. A local boundary would contain this.
3. **Stale closure in `handleClose` (ProductPage)** — `effectiveCreative` is used inside `handleClose` but omitted from the `useCallback` deps array (explicitly commented out). This is a latent bug whenever creative data changes.
4. **Responsive breakpoints defined as scattered magic numbers** — `768`, `769`, `960`, `640`, `504`, `560` appear independently in at least five files. A single constants module would make breakpoint changes safe.
5. **Drag-to-dismiss disabled in doc-scroll mode with no alternative** — on iOS mobile, the swipe-down-to-close gesture is entirely suppressed when the look overlay uses document-scroll, leaving only a small X button. Users lose the most natural dismiss affordance on the platform with the most sensitive UX constraints.

## UI Improvements (5 suggestions)

### UI-1 — Competing Back/Close Affordances in Overlays

**Files:** `app/components/LookOverlay.tsx` · `app/components/ProductPage.tsx`

Both overlays render three back/close affordances simultaneously: a top-left corner button (`.look-back-btn` / `.pd-back`), a vertically-centred side-rail button (`.look-back-rail` / `.pd-back-rail`), and a mobile close button via `OverlayChrome`. On mobile this creates visual clutter. The side rail fades in inconsistently and competes with the corner button once the user has scrolled past the hero.

**Suggestion:** On mobile, rely solely on the `OverlayChrome` button and the swipe gesture. On desktop, show the corner button initially and fade in the side rail only after the user scrolls past the hero (current desktop behaviour is reasonable). Remove the third affordance to reduce cognitive load.

### UI-2 — No Visual Feedback on Super-Admin Long-Press Delete

**File:** `app/components/ProductPage.tsx` (save-button long-press handler, ~line 754)

The 650 ms hold-to-delete gesture triggers with only a `navigator.vibrate(30)` call for feedback. There is no visual progress indicator, so a user who accidentally holds the button has no idea a destructive action is approaching.

**Suggestion:** Add a CSS keyframe progress ring (or border-fill animation) that activates on `pointerdown` and completes over 650 ms. Abort and reverse it on `pointerup`. Show it only in super-admin context so it does not affect regular users.

---

### UI-3 — Skeleton Loader Fade Tied to Timeout, Not Paint Event

**Files:** `app/components/LookOverlay.tsx` · `app/components/ProductPage.tsx`

The skeleton fade-out uses a hardcoded ~1400 ms timeout. When a tap-handoff poster loads almost instantly, the skeleton is still visible for a second after the image appears. On a slow network the timeout fires too early and the skeleton disappears before the content is ready.

**Suggestion:** Drive skeleton removal from the first meaningful paint event: listen for the hero `<video>` or `<img>` `onLoad` / `onCanPlay`, and use the timeout only as a 2 s safety fallback.

### UI-4 — ParticleBackground Not Guarded on Mobile in GridView

**File:** `app/components/GridView.tsx` (around line 251, empty-search state)

`LookOverlay` and `ProductPage` both guard `ParticleBackground` behind `!isMobileViewport()`. `GridView` renders it unconditionally on empty-search results. On a low-end mobile device the shader recompilation can cause a noticeable first-frame jank.

**Suggestion:** Apply the same `isMobileViewport()` guard to the `GridView` particle render. If particles are desired on mobile search-empty states, defer them with a `requestIdleCallback`.

### UI-5 — Inline `style={{}}` Objects Scattered Throughout JSX

**Files:** `app/components/LookCard.tsx` (~lines 334–360) · `app/components/ProductPage.tsx` (multiple layout divs)

Numerous layout values (position, inset, dimensions, z-index) are expressed as inline style objects rather than CSS classes. Each render allocates a new object, theme-level overrides cannot reach them, and dark-mode or skin toggling is impossible without JS.

**Suggestion:** Move static layout inline styles to named CSS classes or CSS custom properties. Reserve `style={{}}` for values that are genuinely dynamic at runtime (e.g., `style={{ top: offsetPx }}`).

## UX Improvements (5 suggestions)

### UX-1 — Drag-to-Dismiss Broken in Doc-Scroll Mode

**File:** `app/components/LookOverlay.tsx` (~line 833)

The iOS Safari bottom-toolbar fix switches the look overlay to document-scroll, which hard-disables the entire drag-dismiss handler. On mobile, users can no longer swipe down to close; only the small X button remains.

**Suggestion:** Implement an alternative dismiss gesture for doc-scroll mode — for example, a "pull down to close" that scrolls the document back to `scrollY = 0` and then fires the close callback, or a prominent drag handle at the top of the overlay content that triggers close on a short downward flick.

---

### UX-2 — Desktop Search Input Focuses with a 100 ms Delay

**File:** `app/components/BottomBar.tsx` (~lines 631–679, `setTimeout` focus call ~line 251)

On desktop, clicking the search pill first opens the sheet (`searchOpen = true`), then focuses the input in a `setTimeout(..., 100)`. Users expect to type immediately after clicking a search field; the 100 ms pause is perceptible and breaks the expectation.

**Suggestion:** On desktop, call `inputRef.current?.focus()` synchronously (or in a `requestAnimationFrame`) as part of the open transition, not in a deferred timeout. The sheet animation and keyboard focus can occur in parallel.

---

### UX-3 — "You Might Also Like" Feed Has No End-of-Results Signal

**File:** `app/components/ProductPage.tsx` (nested `ContinuousFeed`, ~lines 1031–1036)

The YMAL feed defaults to 16 items and extends via an IntersectionObserver sentinel. If the user never scrolls far enough, the sentinel never fires and the feed silently ends. There is no "end of results" message, no "load more" button, and no visual cue that the list is complete.

**Suggestion:** Render a short "You've seen it all" message (or a subtle divider) after the last batch. As a fallback for small overlays, add a "Load more" button below the initial batch that fires the same load action as the sentinel.

---

### UX-4 — Follow Badge Redundant on Creator Catalog Page

**File:** `app/components/LookCard.tsx` (`CreatorAvatarFollow` chip, ~lines 436–467)

When look cards are rendered on `CreatorPage` — a page dedicated to a single creator — each card still shows the creator's follow badge. The creator's identity is already dominant in the page header, making the per-card badge redundant and visually noisy.

**Suggestion:** Add a `hideFollowBadge` prop to `LookCard` (or `LookCardCreatorChip`). Pass `hideFollowBadge={true}` from `CreatorPage` so the chip shows the avatar and name but omits the follow toggle.

---

### UX-5 — No Accessibility Landmarks on Overlay Sections

**Files:** `app/components/LookOverlay.tsx` · `app/components/ProductPage.tsx` · `app/components/FeedSection.tsx`

Hero sections, product tabs, and "similar looks" feeds are all plain `<div>` containers. Screen readers cannot identify content regions, and keyboard navigation offers no shortcut to jump between them.

**Suggestion:** Replace key wrapper divs with semantic HTML:
- Hero area → `<section aria-label="Look video">`
- Product list → `<section aria-label="Products in this look">`
- Similar feed → `<section aria-label="Similar looks">`

This is a low-effort change with meaningful accessibility impact.

## Codebase Improvements (10 suggestions)

### CODE-1 — Stale Closure: `effectiveCreative` Missing from `handleClose` Deps

**File:** `app/components/ProductPage.tsx` (`handleClose` useCallback, ~lines 1013–1036)

`handleClose` reads `effectiveCreative` inside its body but the deps array explicitly omits it (the comment acknowledges this). If `effectiveCreative` changes before the user closes the overlay, the handler will reference the stale value.

**Suggestion:** Add `effectiveCreative` to the `useCallback` dependency array. If that causes undesirable re-creation, restructure to capture only the stable IDs and look up `effectiveCreative` inside a ref.

---

### CODE-2 — Unbounded Follow-State Cache

**File:** `app/components/LookCard.tsx` (`useFollowState` usage, ~lines 82–92) · follow-state service module

The shared follow-state cache is never evicted. On a 500-card feed with 200 unique creators the cache retains state for every seen creator for the lifetime of the session.

**Suggestion:** Implement a simple LRU eviction in the follow-state service. Keep the N most-recently-accessed creators (e.g., 64) and drop older entries. Re-fetch on re-entry if evicted; the extra network call is negligible compared to the memory saving.

---

### CODE-3 — Missing Error Boundaries Around Nested ContinuousFeed

**Files:** `app/components/LookOverlay.tsx` (~lines 1430–1446) · `app/components/ProductPage.tsx` (YMAL section)

Neither overlay wraps its nested `ContinuousFeed` in an `ErrorBoundary`. A thrown error in any card or the feed itself propagates to the root boundary, unmounting the entire overlay and losing the user's context.

**Suggestion:** Wrap each nested feed in a local `ErrorBoundary` that renders a "Feed unavailable — tap to retry" fallback. The overlay stays visible and the user can still see the primary content.

---

### CODE-4 — Breakpoints Defined as Scattered Magic Numbers

**Files:** `app/components/LookOverlay.tsx` (~line 127) · `app/components/ProductPage.tsx` (~lines 829, 852) · `app/components/BottomBar.tsx` (~lines 131–132)

Mobile/desktop thresholds appear as raw numbers (`768`, `769`, `960`, `640`, `504`, `560`) in at least five files. Changing a breakpoint requires hunting down every occurrence.

**Suggestion:** Create `app/constants/breakpoints.ts`:
```ts
export const BP = { mobile: 768, tablet: 960 } as const;
export const isMobile = () => typeof window !== 'undefined' && window.innerWidth < BP.mobile;
```
Replace all inline number comparisons with references to `BP`.

### CODE-5 — Duplicate Scroll-Stash Pattern in LookOverlay and ProductPage

**Files:** `app/components/LookOverlay.tsx` (~lines 787–819) · `app/components/ProductPage.tsx` (~lines 983–1011)

Both files contain two effects that do the same thing: a `useLayoutEffect` to restore a saved scroll position, and a `useEffect` to continuously record the live scroll offset. The logic is identical.

**Suggestion:** Extract to `app/hooks/useScrollStash.ts`:
```ts
export function useScrollStash(scrollRef: RefObject<HTMLElement>, key: string): void
```
Use it in both overlays. One change point, zero drift.

### CODE-6 — Director Scope Collision in Nested Look + Product Overlays

**Files:** `app/components/LookOverlay.tsx` (`directorScope`, ~line 708) · `app/components/ProductPage.tsx` (`directorScope`, ~line 529)

When a product is opened from inside a look overlay, both scopes are active. The nested YMAL `ContinuousFeed` in each overlay uses the same `slotPrefix` convention, which can register the same `look_id` under two different director scopes and cause unexpected video attach/detach transitions.

**Suggestion:** Ensure `slotPrefix` is globally unique by appending the overlay role:
- Look: `look:${look.id}:ymal`
- Product: `product:${brand}:${name}:ymal`

Or assign each overlay instance a stable UUID on mount and use that as the prefix.

### CODE-7 — `setTimeout` Animation-Out Instead of `transitionend`

**File:** `app/components/LookOverlay.tsx` (close handler, ~lines 751–773)

The close path delays `onClose()` by 360 ms to let the CSS exit animation complete. If the transition duration ever changes in CSS, the JS timeout silently falls out of sync, causing either a flash-of-unmount or a noticeable pause.

**Suggestion:** Use `element.addEventListener('transitionend', onClose, { once: true })` with a 500 ms safety timeout instead of the fixed 360 ms. This couples the unmount to the actual animation end rather than an arbitrary constant.

### CODE-8 — No ESLint Rule Blocking `console.log` in Source

**File:** `vite.config.ts` (terser `drop: ['console']` in prod build)

`console.log` is silently stripped in production builds. This is correct, but it means a PR can land with debug logs, they will pass CI, and the mistake is invisible until someone checks the source. It also means log-based debugging is lost for production investigations.

**Suggestion:** Add an ESLint rule:
```json
"no-console": ["warn", { "allow": ["warn", "error"] }]
```
This surfaces accidental console statements in the editor and in CI lint runs without blocking legitimate `console.warn` / `console.error` usage.

### CODE-9 — Unbounded `ContinuousFeed` Card Count (No Virtualisation)

**File:** `app/components/FeedSection.tsx` · `app/components/ContinuousFeed.tsx`

On a long infinite-scroll session (200+ cards), every `CreativeCardV2` instance remains mounted with its own `IntersectionObserver`, `useTrailVideo` hook, and event listeners. The director's slot registry grows linearly. On lower-end devices this accumulates memory and degrades scroll performance.

**Suggestion:** Introduce windowing. Either use `react-window` / `react-virtual` or implement a lightweight custom approach that unmounts cards that scroll more than ~1000 px outside the viewport, reusing slots on re-entry. This is the single highest-leverage performance improvement for the feed.

---

### CODE-10 — Optional Callback Props Called Without Null Guard

**Files:** `app/components/LookOverlay.tsx` (~line 173) · `app/components/ProductPage.tsx` (~line 1398)

`onDailyFeedBar` is typed as optional but called as `onDailyFeedBar(reached, hidden)` without a null check. If a parent component omits the prop, this throws at runtime. By contrast, `onHome` is correctly guarded with `onHome ?? handleClose`.

**Suggestion:** Apply optional chaining consistently:
```ts
onDailyFeedBar?.(reached, hidden);
```
Audit every optional prop in the overlay/page component signatures and ensure all call sites use `?.()`.
